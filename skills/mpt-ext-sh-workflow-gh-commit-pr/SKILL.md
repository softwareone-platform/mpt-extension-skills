# GH Commit PR

Use this workflow to make Git commits and manage PRs consistently with repository rules.

## Workflow

1. Read repository policy first.
- If `AGENTS.md` exists at repo root, follow it as source of truth.
- If missing, use safe defaults: Conventional Commit subject, short PR description, no force-push unless requested.

2. Prepare a clean commit.
- Inspect current branch and staged/unstaged diff.
- Keep scope minimal and avoid unrelated files.
- Build commit message with `<type>: <subject>`.
- Keep subject imperative, under 72 characters, no trailing period.

3. Validate before commit.
- If `AGENTS.md` defines required checks, run those checks exactly.
- If `AGENTS.md` is missing or does not define checks, infer the best minimal check set from repo tooling (for example `make check-all`, `make test`, `npm test`, `pnpm test`, or focused tests for changed scope).
- If check requirements are still unclear or potentially expensive, ask the user which checks to run.
- Never silently skip validation. If checks are skipped by user request or due to constraints, state what was skipped and why.

4. Commit and push.
- Stage only intended files.
- Commit once unless user asked otherwise.
- Push branch and keep existing PR when one already exists for the branch.

5. Create or update PR with `gh`.
- Prefer updating existing PR instead of creating duplicates.
- Build PR title and body from branch naming and repo rules.
- Ensure required warning line and formatting from `AGENTS.md`; only use skill defaults when no repo policy exists.

6. Handle review threads.
- Push fixes first.
- Reply inline to each open thread with only the fix/status for that thread.
- End every GitHub comment with a blank line and this exact footer:

> 🤖 Generated by AI

## Command Patterns

Use these patterns and adapt arguments to the repo:

```bash
# Check branch status

git status --short
git rev-parse --abbrev-ref HEAD

# Commit

git add <paths>
git commit -m "feat: concise subject"
git push

# Discover existing PR for current branch

gh pr list --head "$(git rev-parse --abbrev-ref HEAD)" --json number,url,title

# Create PR

gh pr create --title "<ISSUE KEY>: <description>" --body-file /tmp/pr_body.md

# Update PR title/body

gh pr edit <number> --title "<ISSUE KEY>: <description>" --body-file /tmp/pr_body.md

# Review threads / comments

gh pr view <number> --comments
```

## Bundled Resources

- Use `scripts/extract_key.sh` to parse the PR key from a branch named `<pr_key>-<description>`.
- Use `scripts/render_pr_body.sh` to generate a compliant PR body with required warning line.
- Use `references/agents-pr-checklist.md` as a quick compliance checklist when the repo has matching rules.

### Script Contracts

- `scripts/extract_key.sh [branch]`
  - Success: prints uppercase PR key to stdout.
  - Failure: exits non-zero with a clear expected-format error.
- `scripts/render_pr_body.sh "<summary>"`
  - Success: prints a PR body including `🤖 **AI-generated PR** — Please review carefully.` and a Summary section.
  - Failure: exits non-zero with a usage error when summary is missing.

## Guardrails

- Never commit secrets or local credentials.
- Never rewrite shared history unless explicitly requested.
- Never close and recreate PRs unless technically blocked.
- Report exactly what was pushed and which PR was created or updated.
